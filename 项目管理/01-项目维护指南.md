# 项目维护指南

> 本文说明如何编写和维护本 PKM 项目。项目规则应显式保存在仓库内,不要依赖某次聊天记录或某个模型的隐藏记忆。

## 1. 项目定位

本项目不是普通资料索引,而是 AI 协作下的个人知识管理系统。它要把一个知识领域从资料调研、主线梳理、可视化解释、实验验证,推进到可长期维护的公开知识板块。

`深度学习/` 是第一个实验板块。后续新增知识领域时,应作为与 `深度学习/` 并列的新板块维护。

## 2. 规则来源与可追溯性

项目约定的优先来源:

1. 当前项目内文档,尤其是 `README.md`、本文、`项目管理/02-迭代记录.md`、`工具轨backlog.md`。
2. 当前项目文件本身,包括各板块的 `知识库/`、`实验/`、`assets/`、`build.py`。
3. 历史会话或外部记忆只能作为补充线索;重要规则必须整理回项目内文档后再作为长期依据。

三条已核实的核心规则:

- **正文与工具想法分离**:未实现的交互、展示、学习工具构想写入根目录 `工具轨backlog.md`,不写入知识正文。
- **一个知识点一个 md**:每个知识点顶部用 `<!--# id -->` 对应板块 `build.py` 的 TREE 节点;文件名和顺序可调整,节点 id 必须稳定。
- **实验先对齐设计、数据代码图统一来源**:实验服务理解,不追求工程完整;正文中的实验图应能由板块内实验代码复现。

## 3. 写作风格

正文面向公开学习者,使用专业、开源、教授式口吻。不要写个人学习状态,不要因为当前学习者已经理解某点而省略关键解释。

一节内容应有清晰主线:先说明问题从哪里来,再说明为什么必须引入某个概念或模型,最后收束到应掌握的要点。不要把概念堆成术语列表。

表达应平实严谨:

- 少用自问自答式标题或段落。
- 少用"长出来""魔法"等生动但不够专业的说法。
- 遇到英文术语,优先给中文解释;英文可作为括注,不能代替解释。
- 本节负责讲清原由和联系;冗长推导放到数学节点或参考链接中承接。
- 应用层节点引用已在本库整理过的数学、统计、线性代数或前置模型概念时,优先链接本库内部节点;不要在正文主线中用 `📖 权威详解` 直接跳出到外部百科。外部权威资料主要放在基础节点或文末参考链接中,避免正文主线绕过本库已有知识结构。该规则适用于同层所有相关节点,不是单篇临时修补。

## 4. 内容组织

每个知识板块采用"源文件与展示分离":

- `知识库/`:内容源头,一个知识点一个 md。
- `assets/`:正文图片和 GIF。
- `build.py`:递归读取 `知识库/`,按 `<!--# id -->` 解析节点,生成 `知识地图.html`。
- `知识地图.html`:阅读视图,不是内容源头。

新增知识点时:

1. 先确定它是否应成为独立节点、子节点,还是并入既有节点。
2. 新 md 顶部写 `<!--# id -->`,并在板块 `build.py` 的 TREE 中登记。
3. 图片放入本板块 `assets/`,源 md 中用相对路径引用。
4. 改完进入板块目录运行 `python3 build.py`,检查 HTML 视图。

## 5. 可视化原则

可视化是本项目的核心能力,不是装饰。图和 GIF 应回答一个具体理解问题,例如:

- 公式或概念的几何含义;
- 训练过程如何变化;
- 两个空间、两个视角或两个模型的差异;
- 某个关键结论为什么成立。

图像要尽量可复现。若图来自实验代码,应在板块 `实验/` 中保留代码、数据和 README;若图是解释性示意图,也要保证文字不把示意图误说成真实计算结果。

## 6. 实验原则

实验是动手轨,服务于理解:

- 开始实验前先对齐目标、数据、要观察的现象、与前序内容的衔接。
- 每个子项目自带 `README.md`、代码、数据和输出图。
- 代码优先用 `# %%` 分单元,便于脚本直跑和 Jupyter / VS Code 逐块运行。
- 文档中的实验图应由同一份数据和代码生成;不要手工改图造成正文与实验不一致。

当前已完成实验在 `深度学习/实验/MLP非线性可视化/`。后续实验主线暂缓到相关原理内容被充分吸收后再推进。

## 7. 编写流程

维护本项目时按以下流程推进:

1. 先阅读根目录 `README.md`、本文和 `项目管理/02-迭代记录.md`,确认当前阶段目标。
2. 若修改某个知识板块,再阅读该板块 `README.md`、相关知识点 md、相邻节点和必要实验说明。
3. 开始前说明本次修改范围。涉及正文大改、节点结构调整、实验设计时,先给预览或方案;确认后再写入。
4. 修改正文时同步维护图片、内链、参考资料和应掌握要点。
5. 修改实验时同步维护数据、代码、输出图和实验 README。
6. 改完进入对应板块运行 `python3 build.py`,检查 `知识地图.html`。
7. 阶段性修改完成后更新 `项目管理/02-迭代记录.md`。
8. 准备提交前检查 Git 状态,用清晰提交信息记录本阶段变化。

小范围错别字、旧口径、断链、渲染兼容问题可直接修复,但应在提交或说明中列出。

## 8. 公开仓库维护检查

每次阶段性提交前至少检查:

- 是否存在依赖作者本机的绝对路径或私有资料路径。
- README 是否能让新读者理解项目定位、结构和入口。
- 各板块 `知识地图.html` 是否能在下载仓库后直接打开。
- 图片、数据、实验代码是否有清晰来源与复现说明。
- 迭代记录是否说明当前状态、挂起事项和下一步。
